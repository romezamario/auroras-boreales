Una visualización científica accesible, trazable y explicable

El sitio integra pronóstico auroral, nubosidad global, condición día/noche y localización aproximada para responder una pregunta operativa: ¿dónde podría verse una aurora con mejores condiciones de observación?

La página reúne en una sola narrativa los fundamentos del proyecto, la forma en que circulan los datos y las decisiones de diseño que convierten información geoespacial compleja en una lectura útil para consulta pública.

1. Contexto del problema

Motivación, objetivos y alcance

Antes de entrar en la implementación, conviene entender qué necesidad atiende la aplicación y qué tipo de respuesta puede ofrecer con los datos disponibles.

Problema de investigación

Las personas interesadas en observar auroras suelen depender de varias fuentes separadas para interpretar actividad geomagnética, nubosidad y oscuridad local. El proyecto reduce esa fragmentación al reunir esas variables en una sola lectura visual, más fácil de consultar y comparar.

Objetivo general

Desarrollar una visualización interactiva de libre acceso que ayude a identificar regiones con mejores condiciones estimadas para observar auroras boreales mediante la integración de actividad auroral y nubosidad.

Objetivos específicos

  • Definir una lógica interpretable para estimar visibilidad a partir de intensidad auroral y nubosidad.
  • Recolectar datos abiertos, periódicos y con cobertura geográfica amplia.
  • Implementar una visualización geoespacial interactiva con efecto de globo 3D sobre canvas 2D.
  • Publicar el resultado como sitio web estático accesible por internet.

Alcance

La solución ofrece una lectura global aproximada, orientada a interpretación visual y apoyo a decisiones. No pretende reemplazar modelos científicos de alta resolución ni estudios atmosféricos locales especializados.

2. Fuentes y datos

Qué información consume el sitio y cómo se transforma

Cada capa visible del globo depende de una fuente distinta. La utilidad del sitio surge precisamente de combinar esas fuentes bajo un mismo modelo de lectura.

Auroras boreales

Se usa el feed OVATION Aurora Forecast de NOAA, consumido en JSON y actualizado de forma casi continua. Cada punto aporta longitud, latitud e intensidad auroral para construir la capa principal del globo.

Aporte a la interpretación: permite estimar dónde existe señal auroral suficiente y filtrar visualmente zonas con mayor relevancia.

Nubosidad global

La capa de nubes se deriva de MODIS/Earthdata y se publica como data/clouds.json, un artefacto ya normalizado para consumo directo desde el frontend.

Aporte a la interpretación: incorpora una restricción atmosférica clave, porque una aurora intensa puede seguir siendo difícil de observar bajo cielos muy cubiertos.

Contexto auxiliar

El sistema incorpora world-atlas para continentes y fronteras, metadata de versión del despliegue y geolocalización aproximada por IP para contextualizar la experiencia del usuario.

Aporte a la interpretación: agrega referencia geográfica, trazabilidad del sitio publicado y un punto de comparación con la ubicación aproximada de quien consulta.

3. Arquitectura

Separación entre runtime web y procesamiento offline

La arquitectura privilegia claridad operativa: el navegador resuelve la interacción y el render, mientras que la preparación más costosa de nubosidad ocurre fuera del runtime público.

Diagrama de arquitectura del sitio

El gráfico muestra cómo el frontend coordina estado, servicios remotos y el artefacto intermedio clouds.json para sostener una experiencia estática pero informativamente rica.

flowchart LR
    U[Usuario] --> H[index.html + style.css]
    H --> APP[js/app.js]
    APP --> STATE[js/state.js]
    APP --> DATA[js/data/*.js]
    APP --> GLOBE[js/globe/*.js]
    APP --> UI[js/ui/*.js]
    DATA --> NOAA[NOAA OVATION]
    DATA --> CLOUDS[data/clouds.json]
    DATA --> IP[ipapi JSONP]
    JOB[scripts/mod08_cloudfraction.py] --> MODIS[NASA LAADS / MODIS]
    JOB --> CLOUDS
    GLOBE --> CANVAS[Canvas interactivo]
    UI --> CANVAS

Frontend modular

  • Presentación: index.html y style.css.
  • Orquestación: js/app.js.
  • Estado compartido: js/state.js.
  • Servicios: js/data/*.js.
  • Render y UI: js/globe/*, js/overlays/* y js/ui/*.

Pipeline offline

scripts/mod08_cloudfraction.py transforma productos MODIS/LAADS en una cuadrícula homogénea y versionable. Esto desacopla la adquisición del dataset de la experiencia interactiva y evita cargar en el navegador tareas pesadas de preparación.

Ventaja de diseño

El contrato intermedio clouds.json hace posible publicar el proyecto como sitio estático simple, portable y económico, manteniendo además claridad entre ingestión, transformación y visualización final.

4. Flujo funcional

Del arranque del navegador al render del globo

El comportamiento observable del sitio depende de un arranque escalonado: primero se prepara la interfaz base y luego se completan en paralelo las capas de datos y el contexto adicional.

Secuencia de carga e interacción

La secuencia resume el orden práctico en que la aplicación inicializa la experiencia, consulta datasets y habilita la primera vista operativa del globo.

sequenceDiagram
    participant B as Browser
    participant App as js/app.js
    participant Refresh as refresh.service
    participant NOAA as NOAA
    participant Clouds as clouds.json
    participant IP as ipapi

    B->>App: DOMContentLoaded
    App->>App: Inicializa UI, canvas y globo
    App->>IP: Solicita localización aproximada
    App->>Refresh: refreshAll()
    par Carga concurrente
      Refresh->>NOAA: fetchLatest()
      Refresh->>Clouds: fetchLatest()
    end
    Refresh-->>App: Actualiza App.state
    App-->>B: requestRender()
  1. Inicialización: el navegador carga la shell visual y prepara canvas, paneles y controles.
  2. Carga base: se descargan masas continentales y fronteras para dar contexto geográfico.
  3. Consulta no bloqueante: la localización por IP y la versión del sitio se obtienen sin retrasar la visualización principal.
  4. Refresco concurrente: auroras y nubosidad se cargan en paralelo con tolerancia a fallos parciales.
  5. Actualización del estado: App.state concentra datasets, umbrales, selección y metadatos.
  6. Interacción: el usuario rota el globo, activa capas, filtra rangos y consulta puntos específicos.
5. Modelo analítico

Reglas de negocio y lógica de visibilidad

El valor explicativo del sitio no está solo en mostrar capas, sino en declarar con claridad qué criterios se usan para ocultar ruido, conservar relevancia y derivar la categoría de visibilidad.

Diagrama de reglas de negocio

Este flujo traduce en términos visuales las reglas descritas en el README.md: filtros de render, degradación parcial y clasificación discreta de la capa Probabilidad.

    G -- No --> G1[La app sigue operando con degradación parcial]
    G -- Sí --> H{¿La nubosidad cae dentro del rango visible?}

    F --> J{¿Se activa la capa Probabilidad?}
    I --> J
    J -- No --> Z6[Se mantiene solo la vista base]

    K -- Sí --> L{¿La ubicación está de noche y en la cara visible?}

    Q --> R

Reglas principales

  • Solo se dibujan auroras dentro del umbral configurado por la persona usuaria.
  • Se reduce ruido fuera de latitudes relevantes para observación auroral.
  • Se omiten puntos ocultos detrás del hemisferio visible.
  • La capa de nubes se filtra por rango porcentual visible.
  • La app sigue operando si falla geolocalización o nubosidad.

Matriz simplificada de visibilidad

El panel de detalle traduce dos variables complejas a una lectura accionable:

  • Alta: intensidad ≥ 70 y nubosidad ≤ 30%.
  • Media: intensidad entre 30 y 60 con nubosidad ≤ 30%.
  • Baja: cualquier otra combinación.

La clasificación no reemplaza el dato original, pero sí reduce la carga interpretativa y vuelve más comprensible la decisión para personas no expertas.

6. Experiencia de usuario

Qué puede revisar la persona usuaria dentro del sitio

La interfaz combina exploración libre, filtros de lectura y paneles de contexto para que la consulta no dependa únicamente de interpretar el color sobre el globo.

Globo interactivo

Representa continentes, auroras, nubosidad y ciclo día/noche sobre un canvas responsive que concentra la atención en la relación espacial entre fenómenos.

Filtros y capas

Permite encender o apagar fuentes y ajustar umbrales de intensidad auroral y porcentaje de nubosidad para adaptar la lectura al interés de cada consulta.

Detalle de puntos

Informa latitud, longitud, intensidad, nubosidad, estado día/noche y probabilidad de visibilidad al seleccionar una ubicación del globo.

Contexto adicional

Incluye localización aproximada por IP y estado de actualización de datos y versión del despliegue, lo que mejora la trazabilidad de lo que se está observando.

7. Seguridad, resiliencia y operación

Fortalezas actuales y condiciones de uso responsable

Al ser un frontend estático, el sitio reduce superficie operativa, pero igualmente exige prácticas claras de privacidad, manejo de secretos y degradación ante fallos externos.

Seguridad y privacidad

  • No hay cuentas ni autenticación, por lo que la superficie de ataque es reducida.
  • La geolocalización es aproximada y basada en IP; no se requiere ubicación precisa del navegador.
  • El token de Earthdata debe permanecer fuera del frontend y limitarse al pipeline offline.

Resiliencia operativa

  • La carga concurrente usa degradación parcial para no derribar la experiencia completa.
  • La máscara día/noche se recalcula localmente, reduciendo dependencia externa continua.
  • El sitio puede publicarse como bundle estático curado, excluyendo artefactos históricos no necesarios.
8. Evaluación crítica

Qué se ha logrado y qué falta para una siguiente etapa

La evaluación crítica permite leer el proyecto no solo como una demo visual, sino como una propuesta técnica con fortalezas actuales y márgenes claros de mejora.

Logros del proyecto

  • Integra fuentes heterogéneas bajo una narrativa visual única.
  • Separa adquisición, estado, render e interfaz con una arquitectura clara.
  • Convierte un dominio científico complejo en una herramienta comprensible para consulta abierta.

Líneas futuras

  • Automatizar pruebas de utilidades, render y validación de artefactos.
  • Migrar gradualmente hacia ES Modules o una separación más estricta de dominio.
  • Reforzar observabilidad, CI/CD y validación periódica de endpoints externos.
  • Evaluar un backend opcional solo si aporta resiliencia real sin perder simplicidad.
