1. Contexto del problema

Motivación, objetivos y alcance

Problema de investigación

Las personas interesadas en observar auroras deben consultar múltiples fuentes para combinar actividad geomagnética, nubosidad y oscuridad ambiental. El proyecto reduce esa fragmentación al unificar dichas variables en una sola experiencia visual.

Objetivo general

Desarrollar una visualización interactiva de libre acceso que ayude a identificar regiones con mayor probabilidad de observar auroras boreales mediante la integración de actividad auroral y nubosidad.

Objetivos específicos

  • Definir una lógica interpretable para estimar visibilidad a partir de intensidad auroral y nubosidad.
  • Recolectar datos abiertos, periódicos y con cobertura geográfica amplia.
  • Implementar una visualización geoespacial interactiva con efecto de globo 3D sobre canvas 2D.
  • Publicar el resultado como sitio web estático accesible por internet.

Alcance

La solución ofrece una lectura global aproximada, orientada a interpretación visual y apoyo a decisiones, no a predicción científica de alta resolución ni a análisis atmosférico local detallado.

2. Fuentes y datos

Qué información consume el sitio y cómo se transforma

Auroras boreales

Se usa el feed OVATION Aurora Forecast de NOAA, consumido en JSON y actualizado de forma casi continua. Cada punto aporta longitud, latitud e intensidad auroral.

Uso en el sitio: filtro por umbrales, render sobre hemisferio visible y cálculo contextual del punto seleccionado.

Nubosidad global

La capa de nubes se deriva de MODIS/Earthdata y se publica como data/clouds.json, un artefacto ya normalizado para consumo directo desde el frontend.

Uso en el sitio: superposición visual de cobertura, cálculo de visibilidad y metadatos de actualización.

Contexto auxiliar

El sistema incorpora world-atlas para continentes y fronteras, GitHub API para mostrar versión visible, y geolocalización aproximada por IP mediante JSONP.

Uso en el sitio: orientación espacial, trazabilidad del despliegue y marcador contextual del usuario.

3. Arquitectura

Separación entre runtime web y procesamiento offline

Diagrama de arquitectura del sitio

Este gráfico resume la relación entre frontend, estado compartido, servicios remotos y pipeline offline de clouds.json.

flowchart LR
    U[Usuario] --> H[index.html + style.css]
    H --> APP[js/app.js]
    APP --> STATE[js/state.js]
    APP --> DATA[js/data/*.js]
    APP --> GLOBE[js/globe/*.js]
    APP --> UI[js/ui/*.js]
    DATA --> NOAA[NOAA OVATION]
    DATA --> CLOUDS[data/clouds.json]
    DATA --> IP[ipapi JSONP]
    JOB[scripts/mod08_cloudfraction.py] --> MODIS[NASA LAADS / MODIS]
    JOB --> CLOUDS
    GLOBE --> CANVAS[Canvas interactivo]
    UI --> CANVAS

Frontend modular

  • Presentación: index.html y style.css.
  • Orquestación: js/app.js.
  • Estado compartido: js/state.js.
  • Servicios: js/data/*.js.
  • Render y UI: js/globe/*, js/overlays/* y js/ui/*.

Pipeline offline

scripts/mod08_cloudfraction.py transforma productos MODIS/LAADS en una cuadrícula homogénea y versionable. Esto desacopla la adquisición del dataset de la experiencia interactiva.

Ventaja de diseño

El contrato intermedio clouds.json hace posible publicar el proyecto como sitio estático simple, portable y económico, manteniendo además claridad entre ingestión y visualización.

4. Flujo funcional

Del arranque del navegador al render del globo

Secuencia de carga e interacción

Mermaid ayuda a visualizar el orden real en que el navegador inicializa la experiencia, obtiene datos y solicita el primer render.

sequenceDiagram
    participant B as Browser
    participant App as js/app.js
    participant Refresh as refresh.service
    participant NOAA as NOAA
    participant Clouds as clouds.json
    participant IP as ipapi

    B->>App: DOMContentLoaded
    App->>App: Inicializa UI, canvas y globo
    App->>IP: Solicita localización aproximada
    App->>Refresh: refreshAll()
    par Carga concurrente
      Refresh->>NOAA: fetchLatest()
      Refresh->>Clouds: fetchLatest()
    end
    Refresh-->>App: Actualiza App.state
    App-->>B: requestRender()
  1. Inicialización: el navegador carga la shell visual y prepara canvas, paneles y controles.
  2. Carga base: se descargan masas continentales y fronteras para dar contexto geográfico.
  3. Consulta no bloqueante: la localización por IP y la versión del sitio se obtienen sin retrasar la visualización principal.
  4. Refresco concurrente: auroras y nubosidad se cargan en paralelo con tolerancia a fallos parciales.
  5. Actualización del estado: App.state concentra datasets, umbrales, selección y metadatos.
  6. Interacción: el usuario rota el globo, activa capas, filtra rangos y consulta puntos específicos.
5. Modelo analítico

Reglas de negocio y lógica de visibilidad

Reglas principales

  • Solo se dibujan auroras dentro del umbral configurado por la persona usuaria.
  • Se reduce ruido fuera de latitudes relevantes para observación auroral.
  • Se omiten puntos ocultos detrás del hemisferio visible.
  • La capa de nubes se filtra por rango porcentual visible.
  • La app sigue operando si falla geolocalización o nubosidad.

Matriz simplificada de visibilidad

El panel de detalle traduce dos variables complejas a una lectura accionable:

  • Alta: intensidad ≥ 70 y nubosidad ≤ 30%.
  • Media: intensidad entre 30 y 60 con nubosidad ≤ 30%.
  • Baja: cualquier otra combinación.

Esta clasificación hace que el sitio sea interpretable para perfiles no expertos sin sacrificar trazabilidad de la decisión.

6. Experiencia de usuario

Qué puede revisar la persona usuaria dentro del sitio

Globo interactivo

Representa continentes, auroras, nubosidad y ciclo día/noche sobre un canvas responsive que prioriza el panel visual principal.

Filtros y capas

Permite encender o apagar fuentes y ajustar umbrales de intensidad auroral y porcentaje de nubosidad.

Detalle de puntos

Informa latitud, longitud, intensidad, nubosidad, estado día/noche y probabilidad de visibilidad al hacer clic sobre el globo.

Contexto adicional

Incluye localización aproximada por IP y estado de actualización de datos y versión del despliegue.

7. Seguridad, resiliencia y operación

Fortalezas actuales y condiciones de uso responsable

Seguridad y privacidad

  • No hay cuentas ni autenticación, por lo que la superficie de ataque es reducida.
  • La geolocalización es aproximada y basada en IP; no se requiere ubicación precisa del navegador.
  • El token de Earthdata debe permanecer fuera del frontend y limitarse al pipeline offline.

Resiliencia operativa

  • La carga concurrente usa degradación parcial para no derribar la experiencia completa.
  • La máscara día/noche se recalcula localmente, reduciendo dependencia externa continua.
  • El sitio puede publicarse como bundle estático curado, excluyendo artefactos históricos no necesarios.
8. Evaluación crítica

Qué se ha logrado y qué falta para una siguiente etapa

Logros del proyecto

  • Integra fuentes heterogéneas bajo una narrativa visual única.
  • Separa adquisición, estado, render e interfaz con una arquitectura clara.
  • Convierte un dominio científico complejo en una herramienta comprensible para consulta abierta.

Líneas futuras

  • Automatizar pruebas de utilidades, render y validación de artefactos.
  • Migrar gradualmente hacia ES Modules o una separación más estricta de dominio.
  • Reforzar observabilidad, CI/CD y validación periódica de endpoints externos.
  • Evaluar un backend opcional solo si aporta resiliencia real sin perder simplicidad.
